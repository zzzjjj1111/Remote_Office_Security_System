# 远程阻断功能使用指南

## 📋 功能概述

管理员可以在Web控制台的"异常告警"页面，对高风险告警点击**"阻断隔离"**按钮，远程断开指定员工的网络连接，实现物理隔离。

---

## 🚀 快速开始

### 1️⃣ 数据库迁移（首次使用必做）

```bash
cd database
python add_execution_status.py
```

预期输出：
```

✅ 迁移成功！可以启动系统测试远程阻断功能。
```

### 2️⃣ 启动后端服务

```bash
cd backend
python app.py
```

### 3️⃣ 启动前端Agent（员工端）

```bash
cd frontend_agent
python collector.py --user 20 --interval 10
```

> **注意**：`--user 20` 表示监视员工ID为20的终端，可根据实际情况修改

### 4️⃣ 启动前端Web（管理端）

```bash
cd frontend_web
npm run dev
```

访问：http://localhost:5173/admin/alerts

## 🧪 测试流程

### 场景1：管理员远程阻断员工网络

#### 步骤：

1. **触发告警**
   - 在员工终端上进行高风险操作（如打开敏感文件、插入U盘等）
   - 或者在前端模拟器中发送高风险行为数据

2. **查看告警**
   - 登录管理端：http://localhost:5173/admin/alerts
   - 找到对应的告警记录，状态应为"待处理"

3. **执行阻断**
   - 点击该告警行的 **"阻断隔离"** 按钮
   - 确认弹窗中点击 **"立即阻断"**

4. **观察效果**
   
   **管理端显示：**
   ```
   ✅ 阻断指令已下发！终端将在10秒内执行网络隔离。
   ```
   
   随后（5-60秒内）：
   ```
   ✅ 终端已成功执行网络阻断！
   ```
   
   **员工端日志（collector.py控制台）：**
   ```
   📥 收到 1 条待执行指令
   
   🚨 收到管理员远程阻断指令 (告警#123)
   
   🚫 【真实执行】Windows 网络隔离
      执行命令: netsh interface set interface "以太网" disabled
      ✅ 以太网已禁用（终端隔离）
   
   ✅ 阻断执行结果已上报: 网络已断开（终端隔离）
   ```

5. **验证阻断效果**
   
   **员工可以尝试的操作：**
   - ✅ 打开PyCharm编辑代码 → 正常工作
   - ✅ 查看本地文件 → 正常
   - ✅ 使用计算器、记事本 → 正常
   
   **员工无法进行的操作：**
   - ❌ 浏览器访问百度 → 显示"无网络连接"
   - ❌ 微信/钉钉发送消息 → 显示"离线"
   - ❌ Git push/pull → 连接超时
   - ❌ 访问公司内网 → 无法连接

6. **查看执行状态**
   
   在管理端告警列表中，"阻断结果"列会显示：
   - ⏳ **执行中** → 指令已下发，等待终端执行
   - ✅ **已执行** → 终端已成功断开网络
   - ❌ **失败** → 执行失败（可能是权限问题）

## 🔍 故障排查

### 问题1：点击"阻断隔离"后，员工端没有反应

**可能原因：**
- Agent未运行或已崩溃
- 网络连接中断，Agent无法轮询指令
- Agent轮询间隔太长（默认10秒）

**解决方法：**
1. 检查Agent是否在运行
2. 查看Agent控制台是否有错误日志
3. 缩短轮询间隔：`python collector.py --user 20 --interval 5`

---

### 问题2：显示"❌ 失败"，但Agent日志显示成功

**可能原因：**
- Agent上报执行结果时网络中断
- 数据库写入失败

**解决方法：**
1. 检查Agent日志中的"结果上报"部分
2. 手动刷新管理端页面，重新查询状态
3. 检查后端日志是否有数据库错误

---

### 问题3：网络阻断后无法恢复

**Windows手动恢复：**
```powershell
# 以管理员身份运行
netsh interface set interface "以太网" enabled
# 或者
netsh interface set interface "Wi-Fi" enabled
```

**macOS手动恢复：**
```bash
sudo ifconfig en0 up
```

**通过Agent恢复（推荐）：**
暂时未实现自动恢复功能，需要管理员手动在员工终端执行上述命令。

## 📊 API接口说明

### 1. 下发阻断指令

**请求：**
```http
POST /api/alerts/{alert_id}/execute-block
Authorization: Bearer {token}
```

**响应：**
```json
{
  "code": 200,
  "message": "阻断指令已下发，等待终端执行",
  "data": {
    "alert_id": 123,
    "user_id": 20,
    "status": "pending"
  }
}
```

---

### 2. Agent轮询待执行指令

**请求：**
```http
GET /api/agent/{user_id}/pending-commands
```

**响应：**
```json
{
  "code": 200,
  "data": {
    "commands": [
      {
        "alert_id": 123,
        "type": "network_block",
        "created_at": "2024-01-15T10:30:00"
      }
    ],
    "count": 1
  }
}
```

---

### 3. Agent上报执行结果

**请求：**
```http
POST /api/agent/command-result
Content-Type: application/json

{
  "alert_id": 123,
  "command_type": "network_block",
  "success": true,
  "message": "网络已断开（终端隔离）"
}
```

**响应：**
```json
{
  "code": 200,
  "message": "结果已记录"
}
```

## 💡 最佳实践

### 1. 阻断前确认

- 仔细查看告警详情，确认是真正的安全威胁
- 检查用户信任分和历史行为记录
- 优先尝试"发送预警"或"限制权限"等温和措施

### 2. 阻断后跟进

- 及时联系被阻断员工，了解情况
- 如果误判，立即恢复网络并标记为误报
- 记录阻断原因和处理结果，用于后续审计

### 3. 批量阻断慎用

- 不建议同时阻断多个员工（可能影响团队工作）
- 优先处理高风险（critical级别）告警
- 对于中低风险，优先使用二次确认机制

## 🎯 技术架构

```
┌─────────────┐         ┌──────────┐         ┌─────────┐
│  管理端Web   │ ──────► │  后端API  │ ◄───── │ 员工Agent│
│             │  POST    │          │  GET    │         │
│ 点击阻断按钮 │  下发指令 │ 数据库    │  轮询   │ 执行阻断 │
│             │          │ 存储状态  │         │ 上报结果 │
└─────────────┘          └──────────┘         └─────────┘
     ↑                        ↑                     ↑
     │                        │                     │
     │                   MySQL数据库                 │
     │                        │                     │
     └────────────────────────┴─────────────────────┘
                  状态同步（5秒轮询）
```

## ⚠️ 注意事项

1. **权限要求**
   - Windows：需要管理员权限才能执行网络阻断
   - macOS：需要sudo权限

2. **网络依赖**
   - Agent必须能访问后端API（127.0.0.1:5000）
   - 如果员工完全断网，将无法接收新指令

3. **恢复机制**
   - 目前暂无自动恢复功能，需手动执行恢复命令
   - 建议在非工作时间测试阻断功能

4. **兼容性**
   - 支持Windows 10/11
   - 支持macOS 10.15+
   - Linux暂未实现
